# AKRAM OS - System Operating Rules

## Purpose
This repository is the operating system for planning, executing, and refining:
- workflows
- quotations
- technical proposals
- content and launch systems

## Priority Order
When handling any request, use this priority:
1. Identify the domain
2. Identify the requested outcome
3. Select the right playbook
4. Use templates before freeform generation
5. Produce structured outputs, not messy brainstorming

## Core Rules
- Do not treat all files as equal context.
- Prefer system rules and domain context before using prompt libraries.
- Never mix quotations logic with proposal-writing logic unless explicitly requested.
- Every task must be classified first as one of these:
  - domain task
  - playbook execution
  - template generation
  - research/support task
- If context is missing, ask for the minimum missing input.
- Default to concise, structured outputs.
- Avoid duplicating assets across multiple folders.
- Suggest file placement when creating any new file.

## Expected Output Style
For every major task, respond with:
1. Objective
2. Required inputs
3. Recommended file/location
4. Execution steps
5. Final output format

## Folder Logic
- core = system rules and foundational decisions
- domains = business/domain-specific context
- playbooks = repeatable operating procedures
- templates = reusable structured output formats
- library = prompts, references, and supporting assets
- state = active work, backlog, and next actions

## Constraints
- Do not invent project structure unless it fits the folder logic above.
- Do not create new folders casually.
- Keep outputs operational, not theoretical.
